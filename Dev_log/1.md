# Fix Collector Modifier Combination Capture Across Backends

## Summary
Update all collector input paths so modifier-held actions are recorded as stateful combinations, not exact-simultaneous key presses. A modifier press opens a hold window; every key, click, drag, double-click, or scroll during that window records the active modifier(s), and the modifier release is suppressed as a standalone action if it was used.

## Key Changes
- Update the native Wayland/Linux path in `ActionEngine`/`InputMonitor`:
  - `Ctrl down -> C -> S -> Ctrl up` emits exactly `ctrl_l+c` and `ctrl_l+s`.
  - `Super down -> O -> Super up` emits exactly `super_l+o`.
  - Used modifiers are not emitted later as standalone `hotkey` actions.
- Update `cross_platform_capture.py` with the same state machine so fallback backends match native behavior where used.
- Attach active modifiers to mouse/scroll actions:
  - `Ctrl + click/drag/double_click/scroll` records the normal mouse/scroll action plus `action_details.keys=[ctrl_l]`.
  - Action type remains `click`, `drag`, `double_click`, or `scroll`.
- Keep the existing task JSON schema:
  - Expanded keys list, e.g. `keys=[ctrl_l,c]`.
  - Modifier-first ordering, then trigger key.
- Fix serialization in `collector.py` so full combos are written, instead of reducing actions to only the released key.

## Test Plan
- Add synthetic tests for both native and Python fallback logic:
  - Sequential combos under one modifier hold: `Ctrl+C`, `Ctrl+S`.
  - Super/meta combo: `Super+O`.
  - Modifier plus click, drag, double-click, and scroll.
  - Plain unmodified typing remains filtered.
  - No standalone consumed modifier actions are produced.
- Rebuild native module and run existing stress/drag tests plus the new keyboard/mouse combo tests.

## Assumptions
- Wayland is covered by the native Linux backend, not only the Python fallback.
- Existing `action_details.keys` schema remains the target output format.
